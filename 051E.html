Welcome ! Let's discuss Interactive Sciences¶

I'd like to thank Karen Donoghue for exposing me; Dan Kolis, to this commercial opportunity. That E-mail was posted on 04 Nov 2021: I'm sorry I didn't answer sooner; After a few hours researching your offering(s) in processes, I had to think about it, then write up this memo.

After a little research I concluded it's a perfectly clear case when sciencey innovation with large scale R&D turns the corner into a technology that is ready to be deployed. Wow !

Attention

Iterative Scopes is pioneering development of AI-driven computational tools in gastroenterology, a field that has not yet benefited from advanced precision medicine technologies. The aim is to improve GI drug development and clinical care. Computer vision is a branch of artificial intelligence that allows computers to analyze and interpret digital images and videos.

Techniques within the field of computer-vision include:

  • ​Detection – Determining when an object is present in an image / video

  • Localization – Defining where exactly an object is with an image / video

  • Segmentation – Classification of where an object is at the pixel level

    The company is applying these techniques to endoscopy videos that assist physicians in comprehensively documenting all the relevant clinical features that exist for a patient.

My perception, (which maybe quite incorrect), is the Cambridge side technology for working with machine vision is far enough along it's time to "get after" implementing the science in a somewhat deployable piece of routine technology. If the algorithms/programs are the crown jewels of the company, then the peripheral aspects need to be completed to make a user ready system.

The notion here is creating a custom device platform to do this in real time is utterly unavoidable. Thast the M.L. and so on needs to be actionable in a clinical setting. Exporting images and post processing is significant, but it doesn't either demonstrate the science component in it's best light or move to a testing/confirmation product cycle. Even a goal of a patent portfolio needs to be reconciled with the solution(s) implemented in more of an appliance modality then a software release using images from other instrumentation in non-real time.

The strong suggestion is that design and fabrication of a single board computer, housing, documentation and liasion for conformance engineering is a good fit for a small company which exists expressly for what electronics industries people call short runs. Specifically, the suggestion is a "real employee" is a step as opposed to achieving a milestone, that to make a real device a facility is required, not a single person. How capable they are, or experienced, etc. is not even particularly relevant.

The need is to get something made--so who actually does that is more of a game changer then how it's decided.

Specifications Considered

How it's specified is a tricky question. In the old days, the notion was that such a specification is a "moon shot" in management terms. Now, open source software and the stability of computer solutions means the specification needs to exceed minimums, not normalize to expected functionality. If, for instance human factors is leaving a practitioner unhappy, this should be easy to improve before the next usage session. Other software support approaches in other fields are instructive. For example: weapons systems programs are frequently now "touched up" between firings ! The idea changing most programs is a big deal, is bad-old-school nowadays, entirely. Getting a few particular things wrong in this process is incredibly serious, but the majority of line items are fixed with a little typing and some clicks. If the minimums were exceeded on the first pass, the fixes are trivial.

Dialog about subsystems on the Single Board Computer (SBC)

The totally above, minus the M.L. A.I. is technology not science. Making the red interface area exist means any future look and feel is easy. There is no R&D. This is about relatively little specification; the functionality is easy to make exceed the medium term needs with ease.

A little more about each subsystem. Again, there is a hardware and software component to each. However endless open source code is available for all except the **probe* subsystem.

HDTV [ interface ]

Two outputs are suggested as a baseline. One for viewing the primary medical activity, and the second for displaying menu like options as a reminder. My baseline notion is that most real time changes are simply spoken; but a reminder of what they are is useful. This would be provisioned with a small screen for the menu and the large one is up to the site to supply/configure.

I/O Probe [ interface ]

The probe subsystem interface is really the deal breaker on justifying a custom board. That is, some 19 inch rack with plug-ins might be enough for some, if not all, business and science goals. But adding in the need for the probes takes it out of the ballpark of a routine solution entirely. The interface should be a separable PCB entirely to accommodate different manufacturer probes and have an ARM9 CPU on the little board with the custom connector(s).

I/O Vital Signs [ Interface ]

Its probably useful to make the diagnostic environment possibly quite stand-alone, so an interface for a cuff for blood pressure and galvanic skin resistance for heart rate is worth provisioning. Of course a software easy option is to disable these. Doing so is not quite trivial, it involves synchronization with other devices.

AUDIO

Audio input from the environment via microphone(s) is essential. It is likely the human interface will lean into the ability to speak keywords. Also, audio transcripts from the Doctor in attendance is obvious. Bluetooth wireless and/or wired with a few common connectors is the obvious implementation.

WORLD

The term "World" is used in some IoT vocabulary to mean both networks and gateways to internet. Of course this device will have some implementation allowing remote access which is much like being on site with the patient, as well as replays of various kinds.

Warning

Slick, easy to change multi-lingual machine interfaces that create a device that allows remote access, forensic traceability and passes international approvals can be a troubling and even deceiving process. The ultimate quality of solutions is application of the understanding of the diseases, interventions and injection of common sense based on the patient. Doing this packaging of that medical science with great computer science, does not make a perfect, or even necessarily adequate solution. The deeper M.L. graphs and ancillary reasoning in the programs, are absolutely pivotal.

Welcome to Computer On a Module (COM) fabrications¶

The reality of making a system with a powerful computer in it "from scratch", no longer exists. All the alternative fabrication ideas re-use prebuilt plug-able components. This makes the system incredibly easier to fabricate, both faster and cheaper to first application. Shown is a reasonably powerful pluggable Computer on Module (COM), which might be a good try for the device considered here. A COM means future upgrades are trivial, new and better modules that are interchangeable are always in the pipeline.

All the incredible delays to train data do not apply to using it. The computer support still has to be fast, but not as remarkably fast as the computers used in development.

However There is a serious complication ! Selecting a standard that fails strands not only the buying decisions but obsoletes the so called custom "carrier board" totally. Whether the units are going to and from trade shows, are in conformance testing with customers or W.H.O. type phased trials, or other critical application situations ... having the standard go out of fashion totally is disruptive and invariably expensive. This occurs over years, not months, but it has some near but not quite: "scrap and rework", of both hardware, software and obsoletes inventories. A change along these lines is sure to have impact at FDA approvals too.

A management process should realize picking this "horse to ride" is a subtle, difficult decision to get right.

Software interface¶

The acceptance procedure for a system "board" for this purposes is based primarily on the software interfaces. Using a COM module, the board will boot up running linux and run programs, that is not the acceptance criteria whatsoever.

I suggest almost all the the firmware s/w entities be part of the board offering, that the addition of the "Science module" is to drop the M.L. A.I. into a system running a well characterized test version that is an already complete placeholder. This is to insure the R&D and conformance process people are not stuck with version control and driver problems.

Clearly, imaging the alimentary canal, processing images and developing helpful directives for people via perfect looking documentation is difficult. But another reason to have a division of responsibility in the system as built versus science add-in's is in the science part, the mapping from images to characterization to decisions, with both people and M.L. A.I. systems, is subject to incremental improvement of a fundamentally impossible to predict nature.

A casual plan for improvements like so: "Only the data will change, the future improvements will not change the running code", is almost never true. So clobbering some of the content of a specific apparatus to upgrade it needs considerable confidence the rest of the thing is software stable.

The goal might be something around 7 submissions / second to the M.L. A.I. subsystem. My general observation is incredibly obvious really; All images submitted should be prioritized, so in principle every frame is a fact to be scanned into the report generation and human review process. The linux only SBC needs a solid state drive and an old fashioned mechanical one. Sizes to be determined. These details are out of the scope of this document, but the hardware must obviously be capable of this trivial support. Of course it's much better to store the backups in R/T offsite, so the encryption needs to be "bulletproof" so it can be poked into the cloud without privacy people problems.

Attention

It's important everyone resist the urge to make the SBC dual boot with some scrounged up windows on it. Security problems are real and as this apparatus moves in the direction of real clinical data, Security matters.

Test Programs are very important¶

Each line item needs rigorous version control. At the earliest "touch" as a process to apply these programs, a forensic trail needs to be constructed for various, ongoing approvals. Each program should have multiple settable criteria, and one for each must be harder to pass then the application. This involves speeds, how it renders images, (sounds, web pages, etc. ), and it must create it's own reports. The product will require rigorous unit tracing. OEG, Scott and Dan have extensive experience with life and death traceability approval software.
